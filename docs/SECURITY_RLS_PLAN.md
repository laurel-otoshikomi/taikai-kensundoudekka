# Supabase Auth / RLS Hardening Plan

作成日: 2026-07-06

## 目的

現在のアプリはフロント側の簡易パスワード判定で運営者/管理者UIを切り替えていますが、DB側のRLSがpublic write寄りのため、将来的な本番運用ではDB側でも権限を強制する必要があります。

この計画は、本番大会データを壊さずに、段階的にSupabase AuthとRLSへ移行するためのものです。

## 現状のリスク

- `tournaments.password` / `tournaments.staff_password` がDBに平文保存されている
- フロントエンドが `tournaments.select('*')` を使うため、匿名ユーザーにパスワード列が返る可能性がある
- `players` / `catches` の insert/update/delete がpublic許可に近い
- 画面上でログインしていなくても、Supabase APIを直接呼べる人はデータを書き換えられる余地がある

## 目標の権限モデル

| ロール | できること |
| --- | --- |
| 参加者/観戦者 | 大会情報、参加者、釣果、順位の閲覧 |
| スタッフ | 釣果登録、釣果編集、釣果削除 |
| 管理者 | 大会設定、参加者管理、ロゴ、順位非表示、大会終了、結果出力、削除 |

## 推奨する移行順

1. `tournament_members` テーブルを追加する
2. Supabase AuthのユーザーIDと大会ID、ロールを紐づける
3. 既存のパスワードログインは一時的に残す
4. アプリ側でSupabase Authログインを追加する
5. RLSを「閲覧public、書き込みmemberのみ」に変更する
6. 本番で十分確認後、平文パスワード列を廃止する

## なぜ一気にやらないか

現在の大会運用中にRLSだけ先に厳しくすると、検量入力や設定保存が突然失敗する可能性があります。特にCloudflare Pagesは静的配信なので、DBポリシー変更とアプリ更新のタイミングがズレると本番障害になります。

そのため、まずテスト環境で以下を確認してから本番へ進めます。

- 匿名ユーザーで順位表が見える
- スタッフで釣果登録/編集/削除ができる
- 管理者で大会設定/参加者管理ができる
- 未ログイン状態でDB直接更新が拒否される

## 実行前チェックリスト

- [ ] `tournament_members` に対象大会とユーザーIDを登録済み
